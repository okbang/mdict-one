User Profiles

 

A user profile is used to identify each user of the application. It includes 2 types of information, one is about the user (Name, password, description), the other is about the rights on Work units granted to the user . User profiles can be accessed for update from the admin area or by clicking on the name of the use logged in in the header of each page.

Creating a User Profile

For example : a new customer called "Hung Ushuld" wants to use the application. In order to let him use the application we need to create his/her account.

First, we need to login the application using an existing account with administration rights, then we will click on the tab Admin and click on User Profiles icon on the left menu.
On the bottom of the list of current application users, click on Add new, then input :

- User ID: This should be a short and easy to remember version of the user name, 
in out example let's input "Hungu". The  User ID is used in the logon screen in the field User ID.

- Password : Please input the chosen password for the user, the password is used in the logon screen in the field password. Be careful, the password is case sensitive.

- User name: The user name is the full name of the user, in the example :"Hung Ushuld"

- Designation: Some free text for you to describe Mr Ushuld

- Status: The status is used for compatibility with dashboard, please choose the most appropriate value corresponding to the user's job (ex: External viewer for Mr Ushuld).

Then we need to grant rights to Mr Ushuld (see below).

Granting rights to a user

Please login the application using an existing account with administration rights, then click on the tab Admin and click on User Profiles icon on the left menu. In the list of users, we must choose the user we need to grant rights to.

Rights are granted on specific projects/groups/organizations only. In our example, Mr Ushuld is the customer of project X, so his rights only apply to project X and nothing else.

In order to express this, in the rightmost column entitles Roles, we will select a suitable role for the user (for more info see Creating a role), and, in the leftmost column called work units, we will select  the "project X" (for more info see Creating a project).

Up to 5 different roles on 5 different work units can be defined, the Role will only apply to the work unit in the same row (exception : see the Note about rights inheritance).

Administration rights

In order to grant Administration rights to a user you must choose a work unit (which work unit is not important) and assign a role to it. This role must have management right on admin home page and sub-pages.

In case a user profile is assigned several roles/Work units; when the user browses the admin home page and sub-pages, the role with the highest right on admin home page will apply.

 

Note about rights inheritance : Since projects are part of groups which are part of organizations( see work units), for example project x belongs to profit center G3 which belongs to company  YellowSoft; the rights granted to a user on an organization will also apply to the groups of the organization, the rights defined for a group will also apply to projects belonging to the group.

For example, if we give the role Customer to Mr Ushuld on the organization YellowSoft, he will be able to access all groups and all projects of YellowSoft.

If a user has 2 conflicting rights, for example role Customer on the organization YellowSoft, and role partner on project project X, the lowest level of work unit will override the others, which means that  Mr Ushuld  will be able to access all groups and all projects of YellowSoft with the  role Customer, except for the project X where the role partner will apply.
